## 2024-07-06 - FileProvider Path Exposure
**Vulnerability:** The application's `FileProvider` configuration in `res/xml/file_paths.xml` exposed the entire internal cache directory by using `<cache-path name="camera_tmp" path="." />`.
**Learning:** Exposing root directories like `.` via `FileProvider` allows any application granted URI access to potentially navigate and read other files within that root, violating the principle of least privilege.
**Prevention:** Always restrict `FileProvider` paths to explicitly scoped subdirectories (e.g., `camera_images/`) and ensure application logic writes shared files into those restricted directories.
## 2024-07-07 - File Deletion Path Traversal Prevention\n**Vulnerability:** The `deleteFile` and `FileCleanupWorker` functions previously accepted raw string paths and deleted files without verifying if the path fell within the application's intended, safe directories.\n**Learning:** Accepting unvalidated file paths for deletion introduces path traversal risks, allowing accidental or malicious deletion of critical files outside the app's internal storage if the input is ever derived from a manipulated source.\n**Prevention:** Always validate file paths by resolving their `canonicalPath` and verifying they begin with safe, permitted directory roots (like `context.filesDir` or `context.cacheDir`) before executing deletion operations.
## 2024-07-08 - Path Traversal via Unsanitized Names
**Vulnerability:** User-provided or imported data (like `profileName` or image file names from a backup JSON) were used directly in `File` path construction without sanitization or directory boundary checks, allowing potential path traversal vulnerabilities.
**Learning:** Even internal app directories can be vulnerable if a file operation traverses out of its intended scope (e.g. escaping `cacheDir` to overwrite files in `filesDir`) using manipulated strings containing `../`.
**Prevention:** Always sanitize input used in filenames (e.g. stripping non-alphanumeric characters) and strictly validate that the resulting `canonicalPath` starts with the intended base directory's `canonicalPath`.
